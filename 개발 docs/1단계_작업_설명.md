# 1단계: Worker 진입점 Pipeline V2 전환 — 작업 설명

**목표:** BullMQ Job을 받았을 때, 지금의 Mock 파이프라인(`videoGeneratorService`) 대신 **실제 AI 파이프라인**(`PipelineOrchestratorV2` + `GeminiProviderAdapter`)을 타도록 바꾼다.

---

## 1. 지금 구조 (전환 전)

```
BullMQ Job (VideoJobData)
    ↓
worker.ts
    ↓
videoGeneratorService.generateVideo(job.data)   ← Mock 흐름
    ├── 안전 검증 (실제: safetyKeywords)
    ├── 장면 프롬프트 생성 (실제)
    ├── veoService.generateScene()              ← Mock (2초 지연 후 가짜 URL)
    ├── mergeScenes()                           ← Mock (버퍼만 저장)
    ├── 자막 생성·저장 (실제)
    └── return { success, videoUrl, subtitleUrl } 또는 { success: false, error }
    ↓
성공 시: notifyBackendVideoCallback(READY, videoUrl, subtitleUrl)
실패 시: notifyBackendVideoCallback(FAILED, errorMessage)
```

- **문제:** 실제 영상 생성·QC·키프레임·Veo 등이 아니라 전부 Mock이라, “한 번에 실제처럼 돌리는” 파이프라인을 타지 않음.

---

## 2. 바꿀 구조 (전환 후)

```
BullMQ Job (VideoJobData)
    ↓
worker.ts
    ↓
VideoJobData → VideoGenerationRequest 매핑
    ↓
PipelineOrchestratorV2.execute(request)   ← V2 파이프라인 (GeminiProvider 사용)
    ├── Book Grounding (도서 팩트)
    ├── Style Bible
    ├── Scene Planning → Script 생성
    ├── 키프레임 생성 (GeminiProviderAdapter → Banana 또는 플레이스홀더)
    ├── 비디오 생성 (GeminiProviderAdapter → Veo 3.1)
    ├── 자막 생성
    └── (Step 7 Assembly는 아직 placeholder — 2단계 FFmpeg에서 연동)
    ↓
반환: PipelineJobResult { status: 'completed' | 'failed', videoUrl?, subtitleUrl?, error? }
    ↓
status === 'completed'
    → notifyBackendVideoCallback(READY, videoUrl, subtitleUrl)
status === 'failed'
    → notifyBackendVideoCallback(FAILED, errorMessage)
```

- **의미:** “영상 생성” 진입점이 Mock이 아니라 **Pipeline V2 한 번**으로 통일되고, Backend 콜백은 그대로 유지된다.

---

## 3. 구체적으로 할 작업

| # | 작업 | 설명 |
|---|------|------|
| 1 | **worker.ts 수정** | `videoGeneratorService` import·호출 제거. `PipelineOrchestratorV2`, `GeminiProviderAdapter` import 및 사용. |
| 2 | **Job → Request 매핑** | `VideoJobData`(bookId, title, author, summary, trigger) → `VideoGenerationRequest`(title, author?, language: 'ko') 변환 함수 작성 또는 인라인 매핑. |
| 3 | **파이프라인 실행** | `GeminiProviderAdapter` 인스턴스 1개 생성, `PipelineOrchestratorV2` 인스턴스 1개 생성 후 `orchestrator.execute(request)` 호출. |
| 4 | **결과 → 콜백 매핑** | 반환값 `PipelineJobResult`의 `status`가 `'completed'`이면 READY + videoUrl/subtitleUrl, `'failed'`이면 FAILED + error. 기존 `notifyBackendVideoCallback` 시그니처 그대로 사용. |
| 5 | **bookId 유지** | 콜백에는 반드시 `bookId`가 필요하므로, Job에서 받은 `job.data.bookId`를 그대로 콜백 인자에 넣기. (파이프라인 결과에는 jobId만 있음) |

---

## 4. 타입 정리 (참고)

- **BullMQ Job 데이터:** `VideoJobData` (shared)  
  - `bookId`, `title`, `author`, `summary`, `trigger`, `retryCount?`
- **파이프라인 입력:** `VideoGenerationRequest` (shared pipeline.types)  
  - `title`, `author?`, `language: 'ko' | 'en'`, `style?`
- **파이프라인 출력:** `PipelineJobResult` (shared pipeline.types, orchestrator에서 VideoGenerationResult로 사용)  
  - `jobId`, `status` ('completed' | 'failed' 등), `videoUrl?`, `subtitleUrl?`, `error?`, 기타

---

## 5. 주의사항

- **Step 7 Assembly:** 지금 V2 파이프라인 안에서는 “Assembly skipped (FFmpeg integration needed)”로 되어 있어, `videoUrl`/`subtitleUrl`이 placeholder 경로일 수 있음. 1단계에서는 “진입점을 V2로 바꾸고, 성공/실패가 콜백까지 correctly 전달되는지”만 검증하고, 실제 파일 경로·FFmpeg 연동은 **2단계(FFmpeg 실제 연동)**에서 진행하면 됨.
- **에러 처리:** `execute()`가 예외를 던지면 현재처럼 catch에서 `notifyBackendVideoCallback(FAILED, message)` 호출하면 됨.
- **기존 동작 유지:** Backend 콜백 형식(READY/FAILED, videoUrl, subtitleUrl, errorMessage)은 그대로 두고, Worker만 내부 구현을 V2로 바꾼다.
